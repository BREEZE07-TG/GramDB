# GramDB Technical Overview

GramDB is a distributed, asynchronous NoSQL-like database system that leverages Telegram channels as a cost-effective, persistent "cold storage" backend while maintaining an in-memory "hot cache" for high-performance querying.

## Core Architecture

### 1. Storage Backend (Telegram)
- **Private Channels**: Data is stored as messages in private Telegram channels.
- **Message Protocol**: Each message contains a JSON-serialized record or a batch of operations.
- **Index Management**: A special "index message" stores the current state of the database structure and pointers to data blocks.
- **Worker Pool**: To mitigate Telegram's aggressive flood limits, GramDB uses a `PyrogramWorkerPool` that rotates through multiple bot tokens.

### 2. Client Library (GramDB)
- **In-Memory Engine**: Uses `EfficientDictQuery` to provide fast lookups and complex filtering on local data.
- **Hydration**: On startup, the client fetches all messages from the Telegram channel to populate its local cache.
- **Persistence Manager**: Handles asynchronous writes. Changes are first recorded in a local Write-Ahead Log (WAL) to ensure data integrity during crashes.
- **Batching**: Operations are batched before being sent to Telegram to optimize performance and minimize API calls.

### 3. Registry API
- **Metadata Storage**: Stores mapping between client keys, Telegram channel IDs, and the current index message ID.
- **Session Leasing**: Ensures that only one "primary" writer is active at a time for a given database, preventing race conditions and index corruption.
- **Heartbeats**: Active clients must send regular heartbeats to maintain their session lease.

## How it Works (The "GramDB Process")

1. **Connection**:
   - Client contacts the Registry API with a `database_url`.
   - API returns the Telegram channel ID and the current `index_message_id`.
   - Client acquires a session lease from the API.
2. **Hydration**:
   - Client uses the worker pool to fetch the index message and all data messages from Telegram.
   - The local `EfficientDictQuery` engine is populated with this data.
3. **Operations**:
   - **Reads**: Performed instantly against the in-memory cache.
   - **Writes**: 
     - Recorded in local `journal.json` (WAL).
     - Applied to the in-memory cache.
     - Enqueued in the `PersistenceManager` for background synchronization to Telegram.
4. **Synchronization**:
   - The `PersistenceManager` batches pending changes.
   - A bot worker sends the batch to the Telegram channel.
   - If successful, the index message ID is updated in the Registry API.

## Registry API Overview

The Registry API is a Quart-based web service that acts as a centralized coordinator for multiple GramDB instances.

- **Storage**: Uses MongoDB to persist metadata and session states.
- **Key Responsibilities**:
  - Managing database registration (`/admin/databases`).
  - Serving metadata to clients (`/metadata`).
  - Handling session concurrency (`/sessions/acquire`, `/sessions/heartbeat`).
  - Locking databases in case of critical errors or unauthorized access.
